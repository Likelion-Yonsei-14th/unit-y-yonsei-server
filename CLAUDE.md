## 브랜치 & 작업 시작 규칙

Claude Code가 이 레포에서 새 브랜치를 만들거나 커밋을 시작하기 전에 반드시 아래 사전 조건을 확인한다.

### 사용자 입력 선행 필수

- **브랜치 생성·커밋 전략은 사용자의 입력값이 선행되어야 한다.** Linear 이슈 번호·브랜치명·커밋 단위 결정은 임의로 진행하지 않고, 사용자에게 먼저 질문해 확인받은 뒤 실행한다.
- auto mode가 활성화되어 있어도 이 규칙은 예외 없이 적용된다. CI·인프라·문서 같은 공통 작업도 동일하다.

### 이슈 선행 체크

- 새 작업을 시작할 때마다 **Linear 이슈가 먼저 생성되었는지 항상 확인**한다. 이슈가 없으면 사용자에게 "Linear 이슈를 먼저 만들어주세요"라고 요청하고, 만들어질 때까지 브랜치 생성·커밋을 진행하지 않는다.
- 이슈 키(`BACK-XX`)와 Function ID를 사용자에게 받아 기록한 뒤 작업을 시작한다.

### 브랜치명 규칙

- 브랜치는 반드시 **Linear 이슈 상세 → "Create branch"로 자동 생성된 이름**을 그대로 사용한다. 포맷: `feature/{identifier}-{title}` (예: `feature/BACK-12-booth-crud`).
- 브랜치는 반드시 **`dev` 브랜치에서 분기**한다 (README 브랜치 전략 준수). `main`이나 다른 feature 브랜치에서 분기하지 않는다.
- 브랜치명은 **반드시 영어**로 작성한다 (README 규칙). Linear 자동 생성 결과에 한글이 포함되어 있으면, 사용자에게 Linear 이슈 상세에서 영어로 수정한 뒤 그 이름을 알려주도록 안내한다.
- `chore/...`, `feat/...` 같은 임의 이름은 사용하지 않는다. Linear ↔ GitHub 양방향 싱크가 깨져 이슈 상태 자동 전이(In Progress / In Review / Done)가 동작하지 않는다.
- 이미 잘못된 이름으로 브랜치를 만들었다면, 사용자에게 알리고 올바른 이름으로 rename하거나 cherry-pick으로 옮긴다.

### 작업 시작 절차

1. 사용자에게 "Linear 이슈가 있나요? 이슈 키와 자동 생성된 브랜치명을 알려주세요"라고 묻는다.
2. 이슈가 없으면 사용자에게 Linear에서 이슈 생성을 요청하고 대기한다.
3. 이슈 키·브랜치명·Function ID를 받은 뒤 해당 브랜치를 체크아웃한다.
   - Linear가 원격 브랜치를 이미 생성한 경우: `git fetch origin && git checkout <브랜치명>` (자동으로 origin/<브랜치명> 추적)
   - 원격에 아직 없는 경우: `git checkout -b <브랜치명> dev` (반드시 `dev`에서 분기)
4. 그 후에 코드 작성·커밋·푸시를 진행한다.

## PR 작성 규칙

Claude Code가 이 레포에서 `gh pr create`로 PR을 생성할 때는 반드시 아래 규칙을 따른다.

## 커밋 & 푸시 규칙

- 수정사항이 발생할 때마다 **논리적 단위별로 커밋을 쪼개서** 작성한다. 여러 성격의 변경(기능 추가 + 리팩토링 +
문서 수정 등)을 하나의 커밋에 뭉치지 않는다.
- 각 커밋은 **만들어지는 즉시 원격 브랜치에 푸시**한다. 여러 커밋을 로컬에 쌓아두고 한 번에 푸시하지 않는다.
- 커밋 메시지는 커밋 컨벤션(`feat:`, `fix:`, `docs:`, `chore:`, `refactor:` 등) 접두사를 사용하고 한국어로 작성한다.
- 푸시 전에 `git status`, `git diff --staged`로 의도한 변경만 포함됐는지 확인한다.

### 템플릿 준수

- `.github/PULL_REQUEST_TEMPLATE.md`의 구조를 그대로 사용한다.
- 섹션 제목(`## 관련 이슈`, `## 작업 내용`, `## Function ID`, `## 변경 사항`, `## 테스트`, `## 리뷰 참고사항`, `## 체크리스트`)을 임의로 변경하거나 누락하지 않는다.
- 체크박스(`- [ ]`)는 실제 해당되는 항목만 `- [x]`로 체크하고, 해당 없는 항목은 그대로 둔다.
- 모든 내용은 **한국어**로 작성한다.

### 섹션별 작성 지침

- **관련 이슈**: `closes #이슈번호` 형식을 유지한다. 이슈 번호는 **반드시 사용자에게 직접 질문해서 확인**한 뒤 채운다. 브랜치명·커밋 메시지·Linear 키(`BACK-5` 등)에서 임의로 추측하지 않는다. 사용자가 "이슈 없음"이라고 명시하기 전에는 PR 생성을 진행하지 않는다.
- **작업 내용**: 이번 PR에서 실제로 한 작업을 불릿으로 간결하게 나열한다. 커밋 히스토리를 기반으로 작성하되 "왜(why)"가 드러나도록 한다.
- **Function ID**: 관련 이슈에서 Function ID를 가져와 기재한다. 사용자가 알려준 이슈 번호로 `gh issue view <번호>`를 실행해 이슈 본문·제목·라벨에서 Function ID(예: `B-01`, `R-02`)를 추출하고, 해당 값을 적은 뒤 체크박스를 `- [x]`로 표시한다. 이슈에서 Function ID를 찾을 수 없으면 사용자에게 다시 확인한다.
- **변경 사항**: 실제 변경 유형에 해당하는 항목만 체크한다.
- **테스트**: 테스트 수행 여부를 정확히 반영한다. 수행하지 않은 항목은 체크하지 않는다.
- **리뷰 참고사항**: 리뷰어가 특별히 확인해야 할 부분이 있을 때만 작성하고, 없으면 섹션은 남기되 내용은 비워둔다.

### PR 제목

- 커밋 컨벤션(`feat:`, `fix:`, `docs:`, `chore:`, `refactor:` 등)을 접두사로 사용한다.
- 70자 이내로 간결하게 작성한다.
- 이슈 키(`BACK-5` 등)를 제목에 괄호로 덧붙이지 않는다. 브랜치명에 이미 포함되어 있어 중복이다.

### 금지 사항

- 템플릿에 없는 섹션(예: `## Summary`, `## Test plan`)을 임의로 추가하지 않는다.
- `🤖 Generated with [Claude Code]` 같은 자동 서명 푸터를 붙이지 않는다 (이 레포 관례).
- PR 본문을 영어로 작성하지 않는다.

### 작성 절차

1. `git log`, `git diff main...HEAD`로 변경 내역 전체를 파악한다.
2. **사용자에게 관련 이슈 번호를 질문**한다. (예: "이 PR에 연결할 GitHub 이슈 번호를 알려주세요. 없으면 '없음'이라고 답해주세요.")
3. `.github/PULL_REQUEST_TEMPLATE.md`를 읽어 최신 템플릿 구조를 확인한다.
4. 템플릿을 복사한 뒤 각 섹션을 채워 HEREDOC으로 `gh pr create --body`에 전달한다.

## 개인 Claude 설정 가이드

`CLAUDE.md`와 `.claude/settings.json`은 **팀 공유 규약**이다. 개인 취향이나 본인만의 워크플로우를 위해 이 파일들을 직접 수정하지 않는다. 대신 아래 로컬 파일을 사용한다 (`.gitignore`에 등록되어 푸시되지 않음).

### 개인 규칙 추가 — `CLAUDE.local.md`

본인만의 추가 지시 사항·메모를 작성한다. Claude Code가 자동으로 `CLAUDE.md`와 합쳐서 읽는다.

예시 용도:
- "응답은 한국어로, 항상 짧게"
- "테스트는 변경 후 자동 실행"
- "특정 라이브러리·패턴 선호"

모든 레포에 공통 적용하고 싶으면 `~/.claude/CLAUDE.md` (홈 디렉토리)에 작성한다.

### 개인 권한 오버라이드 — `.claude/settings.local.json`

본인만 특정 파일을 Claude로 수정·작성하고 싶을 때 사용한다.

예: 본인은 `CLAUDE.md`를 Claude로 수정할 수 있도록 허용

```json
{
  "permissions": {
    "allow": [
      "Edit(CLAUDE.md)",
      "Write(CLAUDE.md)",
      "Edit(./CLAUDE.md)",
      "Write(./CLAUDE.md)"
    ]
  }
}
```

> 공유 `.claude/settings.json`의 deny가 `./CLAUDE.md`와 `CLAUDE.md` 두 형태를 모두 등록하고 있다. 환경에 따라 경로 정규화 방식이 다를 수 있으므로, allow에도 **두 형태를 모두 포함**시켜야 안전하게 오버라이드된다.

### 팀 공유 규약 변경 절차

- `CLAUDE.md` / `.claude/settings.json` 자체를 변경하려면 **반드시 PR을 통해** 팀 합의를 거친다.
- 본인 임시 워크플로우 때문에 공유 파일을 수정·머지하지 않는다.
- 변경이 필요하면 Linear 이슈를 만들고 별도 브랜치에서 PR로 진행한다 (지금 브랜치가 그 사례).
