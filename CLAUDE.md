## PR 작성 규칙

Claude Code가 이 레포에서 `gh pr create`로 PR을 생성할 때는 반드시 아래 규칙을 따른다.

## 커밋 & 푸시 규칙

- 수정사항이 발생할 때마다 **논리적 단위별로 커밋을 쪼개서** 작성한다. 여러 성격의 변경(기능 추가 + 리팩토링 +
문서 수정 등)을 하나의 커밋에 뭉치지 않는다.
- 각 커밋은 **만들어지는 즉시 원격 브랜치에 푸시**한다. 여러 커밋을 로컬에 쌓아두고 한 번에 푸시하지 않는다.
- 커밋 메시지는 커밋 컨벤션(`feat:`, `fix:`, `docs:`, `chore:`, `refactor:` 등) 접두사를 사용하고 한국어로 작성한다.
- 푸시 전에 `git status`, `git diff --staged`로 의도한 변경만 포함됐는지 확인한다.

### 템플릿 준수

- `.github/PULL_REQUEST_TEMPLATE.md`의 구조를 그대로 사용한다.
- 섹션 제목(`## 관련 이슈`, `## 작업 내용`, `## Function ID`, `## 변경 사항`, `## 테스트`, `## 리뷰 참고사항`, `## 체크리스트`)을 임의로 변경하거나 누락하지 않는다.
- 체크박스(`- [ ]`)는 실제 해당되는 항목만 `- [x]`로 체크하고, 해당 없는 항목은 그대로 둔다.
- 모든 내용은 **한국어**로 작성한다.

### 섹션별 작성 지침

- **관련 이슈**: `closes #이슈번호` 형식을 유지한다. 이슈 번호는 **반드시 사용자에게 직접 질문해서 확인**한 뒤 채운다. 브랜치명·커밋 메시지·Linear 키(`BACK-5` 등)에서 임의로 추측하지 않는다. 사용자가 "이슈 없음"이라고 명시하기 전에는 PR 생성을 진행하지 않는다.
- **작업 내용**: 이번 PR에서 실제로 한 작업을 불릿으로 간결하게 나열한다. 커밋 히스토리를 기반으로 작성하되 "왜(why)"가 드러나도록 한다.
- **Function ID**: 관련 이슈에서 Function ID를 가져와 기재한다. 사용자가 알려준 이슈 번호로 `gh issue view <번호>`를 실행해 이슈 본문·제목·라벨에서 Function ID(예: `B-01`, `R-02`)를 추출하고, 해당 값을 적은 뒤 체크박스를 `- [x]`로 표시한다. 이슈에서 Function ID를 찾을 수 없으면 사용자에게 다시 확인한다.
- **변경 사항**: 실제 변경 유형에 해당하는 항목만 체크한다.
- **테스트**: 테스트 수행 여부를 정확히 반영한다. 수행하지 않은 항목은 체크하지 않는다.
- **리뷰 참고사항**: 리뷰어가 특별히 확인해야 할 부분이 있을 때만 작성하고, 없으면 섹션은 남기되 내용은 비워둔다.

### PR 제목

- 커밋 컨벤션(`feat:`, `fix:`, `docs:`, `chore:`, `refactor:` 등)을 접두사로 사용한다.
- 70자 이내로 간결하게 작성한다.
- 이슈 키(`BACK-5` 등)를 제목에 괄호로 덧붙이지 않는다. 브랜치명에 이미 포함되어 있어 중복이다.

### 금지 사항

- 템플릿에 없는 섹션(예: `## Summary`, `## Test plan`)을 임의로 추가하지 않는다.
- `🤖 Generated with [Claude Code]` 같은 자동 서명 푸터를 붙이지 않는다 (이 레포 관례).
- PR 본문을 영어로 작성하지 않는다.

### 작성 절차

1. `git log`, `git diff main...HEAD`로 변경 내역 전체를 파악한다.
2. **사용자에게 관련 이슈 번호를 질문**한다. (예: "이 PR에 연결할 GitHub 이슈 번호를 알려주세요. 없으면 '없음'이라고 답해주세요.")
3. `.github/PULL_REQUEST_TEMPLATE.md`를 읽어 최신 템플릿 구조를 확인한다.
4. 템플릿을 복사한 뒤 각 섹션을 채워 HEREDOC으로 `gh pr create --body`에 전달한다.
